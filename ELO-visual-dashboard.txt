в—Ҹ Data Fetch Pipeline вҖ” How It Works

  The project has two distinct data pipelines that feed the model:

  1. Game Scores (ESPN Site API)

  Module: pipelines/espn_ncaab_fetcher.py
  Script: scripts/fetch_historical_data.py

  ESPN Site API вҶ’ per-team schedules вҶ’ deduplicated games вҶ’ season parquet files

  - Hits
  site.api.espn.com/apis/site/v2/sports/basketball/mens-college-basketball
  - Fetches all 362 teams in one call, then iterates each team's schedule
  endpoint
  - Extracts: game_id (ESPN event ID), date, home/away teams, scores, location
  - Deduplicates (each game appears on both teams' schedules)
  - Saves to data/raw/ncaab/ncaab_games_{season}.parquet
  - Rate: 0.3s delay, ~2 min per season
  - Coverage: 6 seasons (2020-2025), 35,719 games total

  2. Historical Odds (ESPN Core API)

  Module: pipelines/espn_core_odds_provider.py
  Script: scripts/backfill_historical_odds.py

  Season parquet вҶ’ extract event IDs вҶ’ Core API odds endpoint вҶ’ follow $ref
  links вҶ’ odds parquet

  - Reads event IDs from the score parquets above
  - For each event, hits
  sports.core.api.espn.com/.../events/{id}/competitions/{id}/odds
  - That returns a list of $ref URLs (one per sportsbook provider)
  - Follows each $ref to get the full odds payload
  - Parses nested JSON: open/close/current odds for spreads, moneylines, totals
  - Saves to data/odds/ncaab_odds_{season}.parquet
  - Rate: 2 req/sec, ~3 requests per game (1 list + 2 provider refs)
  - Coverage: ~85% of games have odds (small conference games return empty)
  - Checkpoint/resume: Tracks fetched IDs in data/odds/checkpoints/ so
  interrupted runs can continue

  3. Live/Daily Odds (OddsOrchestrator)

  Module: pipelines/odds_orchestrator.py
  Script: scripts/daily_predictions.py

  Fallback chain: TheOddsAPI вҶ’ ESPN Core API вҶ’ ESPN Site API вҶ’ Scraper вҶ’ Manual
  CSV

  - For daily predictions, the orchestrator tries providers in order
  - TheOddsAPI has real multi-book odds but costs credits (500/month free)
  - ESPN Core API is free and unlimited but only has 1-2 sportsbooks
  - Results cached with 5-min TTL, credit budget tracked in
  data/odds_api_usage.json

в—Ҹ Good question. The core problem is that scores and odds are fetched as two
  separate, sequential pipelines вҖ” scores first, then odds as a massive
  backfill. Here's how to fix that:

  Current Pain Point

  Fetch scores (fast) вҶ’ Train model вҶ’ Need odds вҶ’ Wait 15+ hours for backfill вҶ’
  Re-train

  Three Changes to Eliminate the Wait

  1. Combine Scores + Odds Into a Single Fetch

  Instead of two separate pipelines, fetch odds immediately after each game's
  score:

  # Current: two separate scripts
  fetch_historical_data.py   # scores only
  backfill_historical_odds.py  # odds only (hours later)

  # Proposed: one unified pipeline
  fetch_season_data.py  # scores + odds per game, in one pass

  For each game event, the ESPN Site API (scores) and Core API (odds) use the
  same event ID вҖ” there's no reason to wait. Fetch both at once and save a
  single enriched parquet per season.

  2. Nightly Incremental Collection

  Instead of bulk backfills, collect new data daily:

  Scheduled nightly:
    1. Check what games completed today
    2. Fetch scores + odds for those games
    3. Append to season parquet
    4. Data is always ready for training

  This means when you want to retrain or backtest, the data is already there вҖ”
  no waiting.

  3. Speed Up the Remaining Backfill

  For the historical data we still need:

  - Async requests with aiohttp + semaphore (respect rate limit but saturate it)
  - Test higher rate limits вҖ” 2 req/sec is conservative; ESPN Core API may
  tolerate 5-10
  - Skip known empties вҖ” Small conference teams rarely have odds; build a
  skip-list from the 2025 run to avoid wasting requests on 2020-2024

  Recommended Architecture
