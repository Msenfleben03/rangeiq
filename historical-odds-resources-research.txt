  # Schema Drift Resolution Pattern

  **Extracted:** 2026-02-13
  **Context:** When code writes to a database or calls object methods, and the
  actual schema/API doesn't match what the code assumes.

  ## Problem
  Code is written against a documented or assumed schema, but the actual
  database/object has a different structure. This manifests as:
  - `sqlite3.OperationalError: table X has no column named Y`
  - `AttributeError: 'Object' has no attribute 'method_name'`
  - Test fixtures passing but production failing (or vice versa)

  Common causes:
  - Schema was defined in setup scripts but CLAUDE.md/docs show a different
  version
  - Code was written against a planned schema, DB was created with a different
  one
  - Test fixtures create their own tables with hardcoded schemas that drift from
   production

  ## Solution

  **Before writing DB insert/update code:**
  1. Check the ACTUAL schema: `PRAGMA table_info(table_name)` or `SELECT sql
  FROM sqlite_master WHERE name='table_name'`
  2. Check unique constraints — they determine ON CONFLICT behavior
  3. Match column names AND order to the real schema

  **Before calling object methods:**
  1. Grep for `class ClassName` and verify the method exists
  2. Check for `to_dict()` vs `to_json()` vs `save_json()` — common naming
  variations

  **After fixing production code, ALWAYS check test fixtures:**
  1. Search for the table name in test files: `grep -r "CREATE TABLE table_name"
   tests/`
  2. Align test fixture schema with production schema
  3. Run the specific test file to verify

  ## Example

  ```python
  # WRONG: Written against documented schema
  cursor.execute(
      "INSERT INTO team_ratings (sport, team_id, team_name, season, rating_type,
   rating_value) ..."
  )

  # RIGHT: Check actual schema first
  # sqlite> PRAGMA table_info(team_ratings);
  # → (team_id, sport, season, rating_type, rating_value, as_of_date, ...)
  cursor.execute(
      "INSERT INTO team_ratings (team_id, sport, season, rating_type,
  rating_value, as_of_date) ..."
  )

  When to Use

  - Any time code interacts with a database table
  - When calling methods on objects from other modules
  - When test fixtures create their own database tables
  - After ANY schema-related error, check all three: production DB, code, test
  fixtures
